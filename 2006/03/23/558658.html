Getting all you can out of a keyboard layout, Part #0

by Michael S. Kaplan, published on 2006/03/23 03:01 -05:00, original URI: http://blogs.msdn.com/b/michkap/archive/2006/03/23/558658.aspx

Ok, as was previously suggested, I am going to develop a small application that will interrogate a keyboard layout.

The idea is to get all of the information that you can out of it, such that if someone was trying to build a tool like the On Screen Keyboard (OSK), discussed previously, a bit, that they would know what the keyboard has.

Or alternately, if they wanted to write something to make documentation of keyboards? Or any number of other interesting ideas....

(Note -- I am not working from the OSK source here, mainly because I never remembered being too terribly impressed with it in the past!).

I am going to write it in a small C# console application, although you could probably write it in any language you like -- the focus will be on getting the Win32 API to help you find out what is in the keyboard....

So, here goes -- part zero!

Now, to start, I am not going to use the Keys enumeration to get my Virtual Key values. Mainly because I hate it -- attempting to be cute and provide readable names is a useful exercise to a point, but at the point where they take entries from winuser.h that read:

#define VK_OEM_1          0xBA   // ';:' for US

...

#define VK_OEM_2          0xBF   // '/?' for US
#define VK_OEM_3          0xC0   // '`~' for US

...

#define VK_OEM_4          0xDB  //  '[{' for US
#define VK_OEM_5          0xDC  //  '\|' for US
#define VK_OEM_6          0xDD  //  ']}' for US
#define VK_OEM_7          0xDE  //  ''"' for US
#define VK_OEM_8          0xDF

...

#define VK_OEM_102        0xE2  //  "<>" or "\|" on RT 102-key kbd.

and convert them in the enumeration (from the 1.1 version of the .NET Framework):

        OemSemicolon  = 0xBA,
        OemQuestion   = 0xBF,
        Oemtilde      = 0xC0,
        OemOpenBrackets = 0xDB,
        OemPipe       = 0xDC,
        OemCloseBrackets = 0xDD,
        OemQuotes     = 0xDE,
        Oem8          = 0xDF,
        OemBackslash  = 0xE2,

I mean, as I pointed out in other posts, the punctuation moves around in all of the keyboards. What better way to act like that the work was done by an English speaking developer working for a US company than to avoid using generic names?

All I can say is thank goodness they decided to be a bit less provincial in the 2.0 version of the .NET Framework, adding:

        Oem1 = OemSemicolon,
        Oem2 = OemQuestion,
        Oem3 = Oemtilde,
        Oem4 = OemOpenBrackets,
        Oem5 = OemPipe,
        Oem6 = OemCloseBrackets,
        Oem7 = OemQuotes,
        Oem102 = OemBackslash,

Since the code I am putting up here will be able to be compiled in either version, I did not want to be locked out of using the more generic names. I also wanted all of the values available to me, not just the ones in the enum. All the easier to work with the code in C or C++ or somewhere else, if you don't have to remember how to translate all the names....

And let me say for the record that you can tell how much of a geek I am by the fact that I looked at the enumeration name KeysEx as Keys - Ex, not as Key sEx!

Lest people think I am picking on the .NET Framework folks alone, I'll point out that if developers all those years ago were not caught up in how cool it would be for the letter A (U+0041Z) to be under a VK value of 0x41, we would have a nightmare of less than half the size when it came to different language keyboard layouts -- because no one would have fekt the need to move VK values around.
